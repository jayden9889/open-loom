## How Loom Works Technically + Open-Web Building Blocks for a Clone (verified July 2026)

### 1. Loom Chrome extension capture path
Loom's browser recorder is a Chrome/Edge extension (`liecbddmkiiihnedobmlmillhodjkdmb`) plus a web recorder. Public teardown + Loom's own product pages confirm the standard browser stack, NOT `chrome.tabCapture`:
- **Screen:** `navigator.mediaDevices.getDisplayMedia({ video: true, audio: true })` → returns a `MediaStream`; Chrome shows its native "Chrome Tab / Window / Entire Screen" picker. This picker is mandatory and cannot be pre-selected or automated ([Chrome for Developers, screen-capture](https://developer.chrome.com/docs/extensions/how-to/web-platform/screen-capture); [Recall.ai tabCapture guide](https://www.recall.ai/blog/how-to-build-a-chrome-recording-extension)).
- **Webcam + mic:** `navigator.mediaDevices.getUserMedia({ video:{...}, audio:{ echoCancellation:true, noiseSuppression:true } })`.
- **Compositing / encoding:** tracks are combined and fed to a single `MediaRecorder`, which emits a `.webm`. Data arrives in chunks via `mediaRecorder.ondataavailable → recordedChunks.push(event.data)`; `MediaRecorder.start(timeslice)` (e.g. every few seconds) forces periodic `dataavailable` events so chunks stream out during recording rather than only at stop ([BigBinary Loom-alternative build](https://www.bigbinary.com/blog/build-web-based-screen-recorder-loom-alternative-part-1); search-confirmed MediaRecorder/webm behaviour).
- **tabCapture is the alternative Loom does NOT need for its UX:** `chrome.tabCapture` can target a specific tab without the picker and unifies tab-audio + mic, but only works inside extension contexts and only for a *tab* (`chromeMediaSource:'tab'`, `googDisableLocalEcho:false`), captured via `getUserMedia` with those chromeMediaSource constraints, and must run in an **offscreen document** under Manifest V3 because the MV3 service worker has no DOM and can't hold `getUserMedia`/`MediaRecorder`; the stream also dies if the source tab reloads ([Recall.ai](https://www.recall.ai/blog/how-to-build-a-chrome-recording-extension)). Extensions using tabCapture CANNOT record desktop apps (Zoom/Teams/Slack native), multiple tabs at once cleanly, or Firefox (no equivalent API).

### 2. Loom desktop app capture (macOS/Windows Electron-class app)
The desktop app exists because a browser extension cannot capture the *whole OS* plus reliable *system audio*. Native path uses OS screen-capture APIs. On macOS the hard constraint is system (loopback) audio:
- Historically impossible in Chromium/Electron on macOS without installing a **virtual audio driver** (BlackHole, formerly Soundflower) routed as a virtual input device — "a terrible thing for the end user" ([Electron issue #47490](https://github.com/electron/electron/issues/47490); [Recall.ai system-audio](https://www.recall.ai/blog/how-to-get-access-to-system-audio)).
- **macOS 13+** ships **ScreenCaptureKit**, the first Apple API to capture desktop audio with NO signed kernel extension. **macOS 14.2+** adds **Core Audio taps** (per-app + all-app system audio with user permission). Both are now the sanctioned no-driver route.
- Caveat that shapes the clone's UX: **ScreenCaptureKit audio is not isolated** — notifications/music/other media playing during capture land in the recording ([Recall.ai](https://www.recall.ai/blog/how-to-get-access-to-system-audio)).

### 3. Electron 2026 status (directly relevant to a clone built on Electron)
- Electron's `desktopCapturer` now wires ScreenCaptureKit for macOS loopback audio ([Electron #47490](https://github.com/electron/electron/issues/47490), [PR #47493](https://github.com/electron/electron/pull/47493)).
- **As of Electron v39.0.0-beta.4, Chromium made Apple's Core Audio Tap API the default for desktop audio capture, with NO fallback** to the older Screen & System Audio Recording permission path if tap creation fails — so a clone must target macOS 14.2+ for reliable system audio or ship a driver fallback for older OSes.
- Bottom line for a clone: **In-browser (getDisplayMedia) you CAN capture tab/window audio and, on Chrome, "entire screen" system audio via the picker's "share audio" checkbox; you CANNOT reliably capture arbitrary macOS system audio without ScreenCaptureKit/Core-Audio-tap (native, macOS 13/14.2+) or a virtual driver.** Windows loopback (WASAPI) is comparatively easy; macOS is the blocker.

### 4. Webcam-over-screen compositing (the "bubble")
Two approaches; a clone should support both:
- **CSS overlay (Loom's live UX):** screen fills the tab, a circular webcam `<video>` element is positioned bottom-left via CSS — nothing is composited at record time; the bubble is a separate element. Simplest, but produces two independent streams unless burned in.
- **Canvas burn-in (single-file output):** draw screen `<video>` + webcam `<video>` onto a `<canvas>` each frame (`drawImage`, circular clip via `ctx.arc`+`clip`), then `canvas.captureStream(fps)` → `CanvasCaptureMediaStreamTrack`, combine with audio tracks into a new `MediaStream`, record that ([MDN captureStream](https://developer.mozilla.org/en-US/docs/Web/API/HTMLCanvasElement/captureStream); [Nearform screen+webcam mixing](https://formidable.com/blog/2022/screen-webcam-mixing-recording/); [Chrome capture-stream](https://developer.chrome.com/blog/capture-stream)). Pitfalls: `requestAnimationFrame`/`requestVideoFrameCallback` callbacks stall in fullscreen (workaround: `setTimeout` pump); cross-origin images taint the canvas and break capture (needs CORS on the CDN).
- **WebCodecs path (2026 high-perf option):** `VideoFrame` from each source → draw/composite → `VideoEncoder` (hardware H.264/VP9/AV1) → mux yourself. Gives frame-accurate control and offloads from the single-MediaRecorder pipeline, at the cost of writing your own muxer. Not required to match Loom, but the modern efficient route.

### 5. Codecs & containers (Chromium/Electron 2026)
- **Default reliable output = WebM.** `MediaRecorder` on Chromium/Firefox universally supports `video/webm; codecs=vp8,opus` (most compatible), plus `vp9` and (newer) `av1`; audio = Opus ([Recall.ai](https://www.recall.ai/blog/how-to-build-a-chrome-recording-extension); tutorialpedia MIME list).
- **MP4/H.264 recording is now possible but conditional:** Chrome only writes MP4 (`video/mp4;codecs=avc1.*,mp4a.40.2`) **when the OS provides a hardware/OS encoder** — MP4 container support shipped ([chromestatus 5163469011943424](https://chromestatus.com/feature/5163469011943424)). **Always gate on `MediaRecorder.isTypeSupported('video/mp4;codecs=avc1.42E01E,mp4a.40.2')` before use** — a `true` only means the MIME is accepted, not that the encoder runs at the target bitrate on that device ([MDN isTypeSupported](https://developer.mozilla.org/en-US/docs/Web/API/MediaRecorder/isTypeSupported_static)). Practical clone strategy: record WebM/VP8/VP9+Opus for capture reliability; transcode to H.264/MP4 + HLS server-side for playback compatibility.

### 6. "Instant share while still uploading"
Loom returns a share URL the instant recording starts/stops, before processing finishes. Mechanism (Loom uploads to AWS immediately; multi-quality/adaptive versions + thumbnail + transcript generated in background, "30s to a few minutes"; [Atlassian upload docs](https://support.atlassian.com/loom/docs/upload-or-import-videos/), [Atlassian share docs](https://support.atlassian.com/loom/docs/share-your-recording/)):
- The video **ID/URL is minted up front** (deterministic ID → page shell renders "processing").
- `MediaRecorder` `timeslice` chunks are **streamed as they're produced** to object storage — the canonical implementation is **S3 multipart upload via presigned URLs** (client PUTs each part, backend completes the multipart), the resumable/chunked pattern all Loom-clone write-ups converge on ([S3 multipart presigned guide](https://codewithmukesh.com/blog/upload-large-files-aspnet-core-s3-multipart-presigned-urls/); capme.app Loom-alternative teardown). No specific Loom engineering blog on this is public — treat the "S3 multipart + presigned + up-front ID" as the well-documented industry pattern, not Loom-confirmed internals.
- Viewer opening the link early sees the processing state, then the playable stream once the first renditions land.

### 7. Playback + transcription (public sources thin — flagged)
- **Playback:** Loom generates "multiple quality versions for adaptive streaming" ([capme.app teardown]), i.e. **HLS-style adaptive bitrate** off server-side transcode (typical: AWS MediaConvert / ffmpeg → H.264 renditions + HLS manifest). Loom does not publish its exact playback stack; HLS/ABR is the sourced-but-inferred conclusion.
- **Transcription:** background pipeline produces transcript + auto-captions + AI chapters/titles (Atlassian docs confirm the *features* exist). The specific ASR engine is not public; a clone would use Whisper (self-host) or a cloud ASR (AWS Transcribe / Deepgram) fired on the transcode-complete event.

### 8. Non-seekable WebM duration fix (must-have for a clone)
`MediaRecorder` WebM output has **no Duration and often no Cues/SeekHead** → the file won't seek and shows `duration = Infinity` in players ([Chromium bug 40482588/642012](https://issues.chromium.org/issues/40482588); [addpipe writeup](https://blog.addpipe.com/duration-in-webm-videos-produced-by-chrome/)). Fixes:
- **`fix-webm-duration`** (yusitnikov) — pure-JS, appends the missing Duration metadata to the blob, returns a fixed blob; lightweight, client-side ([GitHub](https://github.com/yusitnikov/fix-webm-duration)).
- **`ts-ebml`** (legokichi) — rewrites SeekHead/Seek/Cues + Duration to make it fully seekable; `tools.makeMetadataSeekable` — but slow (~1 min on a 2.5GB file) ([ts-ebml issue #2](https://github.com/legokichi/ts-ebml/issues/2), [#27](https://github.com/legokichi/ts-ebml/issues/27)).
- **`fix-webm-meta`** (StaZhu) — ts-ebml-based, handles >2GB files, adds Duration + SeekHead + Cues ([GitHub](https://github.com/StaZhu/fix-webm-meta)).
- **Server-side canonical fix:** remux with **`ffmpeg -i in.webm -c copy out.webm`** (or to MP4) — no re-encode, no quality loss, writes proper metadata. Best for a clone since it already transcodes server-side.

### Build recommendation for Open Loom
Extension/web recorder: `getDisplayMedia` + `getUserMedia` + single `MediaRecorder(webm;vp8/vp9,opus)`, `timeslice` chunking, CSS bubble for live + optional canvas burn-in. Desktop app (Electron): required for full-OS + macOS system audio; target macOS 14.2+ Core Audio tap / ScreenCaptureKit (Electron v39+ default), ship BlackHole fallback for <13. Upload: S3 multipart + presigned URLs, mint share ID up front. Server: ffmpeg/MediaConvert → H.264 + HLS renditions, `-c copy` remux to fix WebM metadata, Whisper for transcript. Always `isTypeSupported()`-gate any MP4 recording.
