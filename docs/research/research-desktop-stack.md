## Open Loom — Recommended stack: **Electron** (main + preload + renderer), with `ffmpeg-static` for post-processing and a transparent frameless always-on-top BrowserWindow for the webcam bubble. Justification, comparison, and sharp edges below. Every claim cited.

---

### The single most important reference: Cap (CapSoftware/Cap)
An MIT/AGPL open-source Loom clone already exists and is the closest possible prior art. Cap is **~20,000 GitHub stars**, a Turborepo monorepo, built on **Tauri v2 + Rust + SolidStart**, explicitly "cuts out Electron overhead entirely," and does exactly what Open Loom wants: screen + camera + mic capture, floating camera window, local Studio-mode recording (full quality, no compression) plus Instant-mode share links, local editing (backgrounds, padding, rounded corners, shadows, cursor effects), self-hostable. (https://github.com/CapSoftware/Cap , https://openalternative.co/cap). **Licensing nuance that matters:** Cap's app code is **AGPLv3** (you cannot lift it into a permissive project), but its capture crates `cap-camera*` and `scap-*` are **MIT** and reusable. Cap uses **custom Rust media crates** (`crates/*`: recording, capture, camera, audio, encoding, rendering, muxing, export) rather than ffmpeg (https://github.com/CapSoftware/Cap). This is the key strategic signal: **Tauri gives the low-CPU native result, but only because a funded team wrote a full custom Rust media pipeline.** That effort is the crux of the stack decision.

### Option A — Tauri v2 (what Cap uses)
- **Screen capture:** No web `getDisplayMedia` in the Rust webview by default; you capture natively via the **`scap`** crate (same org as Cap) — Rust, native APIs: **macOS ScreenCaptureKit, Windows Windows.Graphics.Capture, Linux PipeWire**. Maturity: **621 stars, beta `v0.1.0-beta.1` (4 Aug 2025), 453 commits, 7 contributors.** Critically, **scap does NOT capture system audio and does NOT capture webcam** — it is screen frames only (https://github.com/CapSoftware/scap).
- **Webcam:** needs a separate native crate — `nokhwa` (cross-platform camera) or Cap's own `cap-camera*`. Another integration + threading + format-conversion surface.
- **Audio (mic + system):** another crate — `cpal` for device I/O, plus native system-audio loopback plumbing you write yourself.
- **Muxing/encoding:** you own it — raw frames → encoder → container. Cap wrote custom encode/mux/render crates precisely because there is no batteries-included path.
- **Windowing/shortcuts/tray:** fully supported — transparent + always-on-top windows, `tauri-plugin-global-shortcut`, `tray-icon` (https://v2.tauri.app/learn/window-customization/ , https://v2.tauri.app/plugin/global-shortcut/ , https://v2.tauri.app/learn/system-tray/).
- **Verdict:** Produces the best end product (native perf, low CPU, small binary) but demands **assembling and synchronising 4+ native Rust crates (scap + nokhwa + cpal + a muxer) with A/V sync, timestamping, and encoder wiring you build yourself.** This is a multi-crate Rust media-engineering project, not a UI project. Highest quality ceiling, highest effort, most sharp edges for an agent-built clone.

### Option B — Electron (recommended)
- **Screen capture:** first-class via **`desktopCapturer`** + `getDisplayMedia` + `setDisplayMediaRequestHandler`; records full screen continuously; `MediaRecorder` writes the file. `MediaRecorder` is a solved, 94.79%-global-support API (https://www.electronjs.org/docs/latest/api/desktop-capturer , https://caniuse.com/mediarecorder).
- **System audio (the historically hard part) — now largely solved natively:** As of **Electron 39.0.0 (28 Oct 2025, Chromium 142, Node 22.20)**, Chromium's **Apple CoreAudio Tap API is the default** for desktop audio capture; you must add **`NSAudioCaptureUsageDescription`** to Info.plist or `desktopCapturer` silently yields a dead audio stream (https://www.electronjs.org/blog/electron-39-0). For Electron 31–39 the community package **`electron-audio-loopback` (118 stars, Electron ≥31.0.1)** bridges it; under the hood the Chromium flags are `MacLoopbackAudioForScreenShare`, `MacSckSystemAudioLoopbackOverride` (ScreenCaptureKit) and `MacCatapSystemAudioLoopbackCapture` (CoreAudio), pass `"loopback"` as the audio param in `setDisplayMediaRequestHandler`; **works macOS 13+/13.2+** (https://github.com/alectrocute/electron-audio-loopback , https://alec.is/posts/bringing-system-audio-loopback-to-electron/ , https://github.com/electron/electron/issues/47490). An even cleaner mic-free path is **CoreAudio Taps via AudioTee.js** (Swift wrapper, macOS 14.2+, needs only "System Audio Recording Only" permission, no app restart, delivers raw PCM, ~625KB binary) (https://stronglytyped.uk/articles/recording-system-audio-electron-macos-approaches).
- **Webcam bubble (draggable, circular, always-on-top over the whole screen):** the canonical Electron pattern — a second **`BrowserWindow` with `frame:false, transparent:true, alwaysOnTop:true`**, `setAlwaysOnTop(win,'screen-saver')`, `setVisibleOnAllWorkspaces`, `-webkit-app-region:drag` + CSS `border-radius:50%` for the circle; camera feed is a plain `<video>` from `getUserMedia`. No compositing math, no per-frame canvas work — the OS composites the window. This is strictly easier than the PWA canvas approach.
- **Muxing / webm→mp4:** bundle **`ffmpeg-static` (1.4k stars, ffmpeg 6.1.1, mac x64+arm64/win/linux binaries, last release 14 Nov 2025)** and remux/transcode after recording (https://github.com/eugeneware/ffmpeg-static). **Codec caveat:** Chromium `MediaRecorder` typically emits VP8/VP9 in WebM; VP9→MP4 is a **re-encode, not a cheap remux**. If you request `video/mp4;codecs=h264` (supported in recent Chromium) or H.264-in-WebM you can remux (stream-copy) cheaply. Plan the encoder choice up front.
- **Global shortcuts / tray:** `globalShortcut` + `Tray` are core Electron modules (start/stop hotkey, menubar app) — parity with Loom's tray control.
- **Verdict:** Slightly heavier binary and higher idle CPU than Tauri, but the **entire capture pipeline is web APIs you already know, system audio is now native, and the webcam bubble is an OS-composited window.** Fastest path to a faithful Loom-UX clone with the fewest bespoke-native landmines.

### Option C — Pure-browser PWA (rejected)
- **Screen:** `getDisplayMedia` records full screen continuously — fine.
- **Fatal gap 1 — user-chosen save folder:** **File System Access API (`showSaveFilePicker`/`showDirectoryPicker`) is Chrome/Edge 105+ ONLY — not Safari, not Firefox; 28.56% global** (https://caniuse.com/native-filesystem-api). On macOS the default browser is Safari, so "let the user pick a save folder" is impossible in a PWA there. You'd be stuck with download-blob UX, not Loom's library.
- **Fatal gap 2 — webcam bubble:** a web page cannot create an OS-level always-on-top overlay window that floats over *other apps* during recording. You can only composite the camera onto a canvas **inside the recorded frame** (per-frame `drawImage` at 30/60fps → measurable CPU/GC cost, and it's baked into the video, not a live draggable overlay the user repositions over their whole desktop).
- **Verdict:** cannot replicate Loom's floating bubble or user-folder library cross-browser. Out.

### Recommendation & rationale
**Build Open Loom on Electron.** Reasoning: (1) Open Loom is a UX-fidelity clone to be produced quickly, not a perf-crown contender — Electron lets the whole record→preview→library flow live in web tech with `MediaRecorder`/`getUserMedia`/`getDisplayMedia`. (2) The one thing that historically pushed screen recorders off Electron — macOS system-audio loopback — **landed natively in Electron 39 (CoreAudio Tap default) and has a mature shim below that**, removing the main reason to pay Tauri's cost. (3) The webcam bubble, global hotkeys, tray, and user-chosen save folder (Node `fs` + `dialog.showOpenDialog`) are all trivial, first-class Electron. (4) Tauri's advantage is real but is *bought* by writing a Cap-style multi-crate native media engine (scap + nokhwa + cpal + custom muxer with hand-rolled A/V sync) — disproportionate effort for a clone, and Cap's app code is AGPL so you can't shortcut by copying it. Keep Tauri+`scap` as the **v2 performance rewrite** if idle CPU/binary size become the differentiator; the MIT `scap`/`cap-camera` crates are the migration path.

### Known sharp edges of the Electron choice
1. **macOS permissions are silent-fail:** without `NSAudioCaptureUsageDescription` (audio) and Screen Recording TCC grant, `desktopCapturer` returns dead/black streams with **no error** (Electron 39 behaviour) — must prompt + guide the user, and permission changes often need an app restart (https://www.electronjs.org/blog/electron-39-0).
2. **Misleading recording indicator:** the Chromium loopback path trips the macOS "screen recording in progress" purple indicator even for audio-only capture (https://stronglytyped.uk/articles/recording-system-audio-electron-macos-approaches).
3. **macOS version matrix:** loopback needs macOS 13/13.2+ (ScreenCaptureKit path) or 14.2+ (AudioTee CoreAudio path); older macOS falls back to no system audio.
4. **webm→mp4 is a transcode if you record VP8/VP9** — choose `video/mp4;codecs=h264` at `MediaRecorder` time (verify with `MediaRecorder.isTypeSupported`) to keep export as a cheap stream-copy; else budget CPU/time for ffmpeg re-encode.
5. **ffmpeg licensing:** `ffmpeg-static` binaries are GPL-built — fine for an open-source app but forces GPL-compatible licensing on Open Loom, or swap to an LGPL/custom ffmpeg build (https://github.com/eugeneware/ffmpeg-static).
6. **Binary size / idle CPU:** Electron ships Chromium (~150–200MB app, higher RAM/idle CPU than the Tauri/Cap baseline) — acceptable for v1, the explicit reason Cap went Tauri.
7. **Transparent always-on-top window quirks:** transparent BrowserWindows have documented platform edge cases (click-through, vibrancy, resize repaint on macOS) — the webcam bubble needs testing across Spaces/fullscreen apps.
8. **Code signing + notarization:** screen/audio capture entitlements require a signed + notarized macOS build for the permission prompts to behave; unsigned dev builds mis-prompt.
