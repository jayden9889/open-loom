All load-bearing technical specifics verified. (3) electron/electron issue #49607 pins v39.0.0-beta.3 as the last working build, placing the CoreAudio-Tap-by-default change at v39.0.0-beta.4 exactly as claimed. (4) Chrome 141 + macOS 14.2 is the floor for getDisplayMedia system audio. Only caveat, non-refuting and editorial: the 'single hardest constraint and the reason a desktop app exists' clause is a judgement, not a fact - system audio is one genuine hard constraint, but screen capture, system tray, and global hotkeys are independent reasons a Loom clone needs a desktop app. Sources: https://www.electronjs.org/blog/electron-39-0 , https://github.com/electron/electron/issues/49607 , https://developer.chrome.com/release-notes/141 , https://developer.apple.com/documentation/CoreAudio/capturing-system-audio-with-core-audio-taps

## CONFIRMED: MP4/H.264 recording via MediaRecorder is now possible in Chrome but only when the OS provides an encoder; you MUST call MediaRecorder.isTypeSupported('video/mp4;codecs=avc1...') before using it, and a true result doesn't guarantee the encoder runs at the target bitrate — so record WebM for reliability and transcode to H.264/MP4+HLS server-side.
Every factual premise checks out. (1) Chrome shipped MP4-container support for MediaRecorder and writes MP4 only when the platform/OS provides a (typically hardware-accelerated) encoder — so MP4/avc1 availability is device- and OS-dependent, not universal (Chrome Platform Status "MP4 container support for MediaRecorder"; Chromium mediarecorder README notes Chrome selects a hardware-accelerated encoder when available). (2) Because support varies per platform/browser, calling MediaRecorder.isTypeSupported('video/mp4;codecs=avc1...') before recording is the correct/required practice (MDN isTypeSupported). (3) A true result is a capability check, not a performance or bitrate guarantee — MDN explicitly warns "Recording may still fail if there are insufficient resources to support the recording and encoding process." The final clause (record WebM for reliability, transcode to H.264/MP4+HLS server-side) is sound engineering advice that follows directly from those facts rather than a verifiable fact itself; WebM (VP8/VP9) is the historically universal, non-OS-encoder-dependent MediaRecorder path in Chrome, so it is the safe default. Nuance: "MUST" is a best-practice imperative, not a spec mandate. Sources: https://chromestatus.com/feature/5163469011943424 · https://chromium.googlesource.com/chromium/src/+/HEAD/third_party/blink/renderer/modules/mediarecorder/README.md · https://developer.mozilla.org/en-US/docs/Web/API/MediaRecorder/isTypeSupported_static

## CONFIRMED: 'Instant share while uploading' = mint the share ID/URL up front + stream MediaRecorder timeslice chunks via S3 multipart upload with presigned URLs while background processing builds adaptive (HLS) renditions, thumbnail and transcript; this is the well-documented industry pattern, not Loom-confirmed internals.
All four sub-components verify as documented, independent industry patterns: (1) MediaRecorder timeslice fires dataavailable every N ms and per-chunk upload is the MDN-recommended approach for long recordings (https://developer.mozilla.org/en-US/docs/Web/API/MediaRecorder/dataavailable_event); (2) S3 multipart upload with a presigned URL generated per part is the AWS-documented browser pattern (https://aws.amazon.com/blogs/compute/uploading-large-objects-to-amazon-s3-using-multipart-upload-and-transfer-acceleration/); (3) minting a share link immediately while transcoding/HLS/thumbnail/transcript run async is the standard modern video-platform architecture (Cloudflare Stream, Mux: https://www.mux.com/articles/build-ai-video-content-processing-for-better-engagement-and-product-experience). The claim correctly and honestly frames this as the industry pattern rather than Loom's confirmed internals. Minor caveat, not a refutation: MediaRecorder chunks after the first are not independently playable (only the first carries the container header), so 'instant share' means the URL exists immediately, not that raw chunks are instantly playable; playback is served by the background-built HLS renditions, which the claim states correctly.

## CONFIRMED: MediaRecorder WebM has no Duration/Cues so it's non-seekable (duration=Infinity); fix client-side with fix-webm-duration (fast, Duration only) or ts-ebml/fix-webm-meta (full seek metadata, slower, >2GB support), or server-side with 'ffmpeg -c copy' remux which is lossless and the canonical fix.
All sub-claims hold against primary sources. MediaRecorder emits WebM with no Duration and no Cues, so browsers report duration=Infinity and cannot seek (Chromium 40482588/642012; Mozilla 1385699, 1068001). fix-webm-duration appends only the Duration element (fast, lightweight; does NOT add Cues/SeekHead) — https://github.com/yusitnikov/fix-webm-duration . ts-ebml reconstructs full seek metadata (SeekHead + Cues + Duration) but parses the whole file in memory; StaZhu's fix-webm-meta is 'based on ts-ebml and support large file (larger than 2GB files)' — https://github.com/StaZhu/fix-webm-meta . ffmpeg -c copy is a lossless remux (no re-encode) and the canonical server-side fix — https://blog.addpipe.com/duration-in-webm-videos-produced-by-chrome/ . Minor build nuance (not a refutation): plain 'ffmpeg -c copy' fixes duration, but for true fast seeking add '-cues_to_front 1' to write Cues at the front, and remux is VP8/VP9-only.

## CONFIRMED: Local-serve tunnels are limited: TryCloudflare quick tunnels are dev-only with ephemeral URLs, a 200-concurrent-request cap and no SSE; Tailscale Funnel is free with a stable public HTTPS URL but non-configurable bandwidth limits and only works while the origin machine is online.
Every sub-claim checks out against primary sources. Cloudflare's official docs state Quick Tunnels are "intended for testing and development only," generate a random subdomain each run (ephemeral URLs), cap at 200 in-flight requests (returning HTTP 429 over the limit), and "lack support for SSE" (https://developers.cloudflare.com/cloudflare-one/networks/connectors/cloudflare-tunnel/do-more-with-tunnels/trycloudflare/). Tailscale docs confirm Funnel is available on the Free plan, gives a predictable stable HTTPS DNS name under your-tailnet.ts.net with auto-provisioned certs, and that "traffic sent over a Funnel is subject to non-configurable bandwidth limits" routed through DERP relays; because Funnel exposes a service running on a tailnet node via that node's tailscaled, it only serves while the origin machine and its funnel are up (https://tailscale.com/kb/1223/tailscale-funnel). One minor nuance worth noting, not a refutation: the exact Funnel bandwidth figure is undisclosed rather than published, and in practice testers report not hitting it even streaming 4K.
